## Background and Motivation

The relevant finding is that each NumPy call (involving one or more ndarray arguments)
adds 0.75 microseconds of overhead, in addition to numerical processing.

When scaled to number of steps per partials, number of partials per piano note,
and total number of piano notes voiced, this per-call overhead can grow to
eclipse the numerical processing time.

This is particularly relevant in real time audio synthesis, where the size of each
callback request (nsamps) is in the low hundreds.

Refer to [[claude/deep_research/2026-07-17-additive-synthesis-gemv-batching.md]]
for full analysis and runnable benchmarks.

## Strategy

Unconditionally process all partials for all piano keys, while allowing nsamps
to decouple from the callback's request size.

The ability to decouple nsamps from max_nsamps comes from the Damped Sine and
the Linear Ramp mathematical model.

Reorganize the pipeline into as few NumPy call as possible, leveraging matrix-
matrix and matrix-vector multiplications to the fullest extent.

Minimize any work-saving logic that has a high dynamic line-of-code execution
count in Python, unless the performance gain can be demonstrated.

## Description of the new pipeline

### Pre-computation for Damped Sine

We prepare a list of all partials from all piano keys. This gives the number
of tracks for this pipeline.

Each partial is described with frequency, decay constant (tau), and relative
amplitude.

The frequencies of fundamentals and partials obey either a 12-EDO or JI model,
and additionally JI model can be customized with a tonic.

Frequency and tau are converted into omega and log_decay. From these, the
`sinexp` and `cosexp` parts are initialized, with `iota` given by
`np.arange(max_nsamps)`.

Once initialized, these parameters and the cached Damped Sine components are
cached for the entire application process.

### Stateful generation for Damped Sine

Upon a key press (attack), the piano note velocity is converted into amplitude,
then converted into per-partial amplitude. Each partial gets assigned a uniform
random initial phase.

The Linear Ramp states and parameters are also initialized for the piano note.

### Summation of partials

Before envelope shaping, the partials that make up each piano key are
summed up. This is performed in one matrix-matrix multiplication step,
via a mixing matrix.

After this step, the number of tracks is now equal to the number of piano keys.

### Proposed pipeline for Linear Ramp

Linear Ramp is implemented with a simple model of a line equation,
```
y = clip ( m * x + c, 0.0, 1.0 )
```
Where `c` is the envelope level at the start of each rendering request,
and `m` is either `m_rise` for attack, or `m_fall` for release.

Where `x` is `iota`, or `np.arange(nsamps)`, as usual.

Re-trigger, and continuation of unfinished rise or fall only affect the value
of `c`, which is carried over from the previous callback request.

Values `m` and `c` remain constant across the `nsamps` dimension. Their values
vary across piano keys, based on the Linear Ramp state of each piano key.

In NumPy, `m` and `c` values are said to be *broadcast* on the `nsamps`
dimension.

`clip` is a single NumPy call.

Envelope shaping is accomplished with an elementwise multiplication between
the Damped Sine (after summation of partials) and the Linear Ramp.

### What holds up

- **All-keys, no allocation.** 88 keys is the physical max polyphony (a key sounds
  one note at a time; re-trigger restarts it), so this retires allocation,
  stealing, `max_voices`, and the phase-discontinuity steal artifact. Correct and
  significant.
- **Two decays kept separate.** Exponential string decay lives in `sinexp/cosexp`
  (tau per partial); the Linear Ramp is a separate, faster gate for attack-click
  suppression and note-off damping. Right call.
- **`clip(m*x+c, 0, 1)` gives sustain for free.** Once the rise line reaches 1.0,
  `clip` holds it flat with no state change; note-off flips `m` to `m_fall`. Minimal
  and correct.

### The caveat that changes a number

**Per-key envelope forfeits the single-GEMV full collapse, so the report's
1.3% / 57x headline does NOT transfer to this pipeline.** That figure was for
collapsing everything to one mono output in a single `ab @ M` GEMV. A per-key Linear
Ramp forces the 88 keys to stay *separate* through the mixdown, which is a strictly
bigger operation:

- Block-diagonal GEMM `C(K, 2P) @ M(2P, N)`: hits BLAS speed but ~99% of the flops
  land on off-diagonal zeros.
- Segment-sum (`ab[:,None]*M` then `np.add.reduceat` over per-key partial ranges):
  no wasted flops, but runs at NumPy elementwise (memory-bound) speed.

Both plausibly land in the few-hundred-us to ~1 ms range for all 88 keys (~10-25%
of the 8685 us deadline) — still a large win over "16 voices ~= 26%", but not 1.3%.
**This per-key mix is the one operation to benchmark before trusting the design** —
it is exactly the "gate construction cost not measured" gap the report flagged.

### On summing only over active-note partials (human's follow-up)

Summing over only the partials of active notes is the right instinct for real
playing (few keys down at once), and it is the natural remedy for the wasted-zeros
problem above — but note it is a **partial walk-back of "unconditionally process all
partials for all keys"** in the Strategy section, and the dynamic bookkeeping it
needs is precisely the "work-saving logic with high dynamic line-of-code count" that
same section says to avoid *unless the gain is demonstrated*. So treat it as a
measured optimization over the fixed-88 baseline, not the starting point.

Two guardrails if you do it:

1. **Dynamic *size*, static *allocation*.** Never reallocate the mixing block-matrix
   in the callback — that reintroduces the allocation latency-tail the report saw
   (median 110 us vs max 394 us). Preallocate at the max (all 88 keys) and operate on
   a variable-length *prefix* of `A` active keys: write the active partials into the
   first `32*A` rows (16 partials x 2 for sin/cos) of a preallocated buffer and
   GEMM / `reduceat` over `[:32*A]`.
2. **Gather is a small contiguous copy.** Active keys are not contiguous in the static
   table (C4 and C6 pressed together), so selecting their partial rows is a fancy-index
   gather. Keep partials grouped by key in `M` (key `k` owns rows `[32k : 32k+32]`) so
   the gather pulls whole key-blocks; the copy is `32*A x N` floats (~123K at A=10),
   cheap, and it also lands the data contiguously — which incidentally sidesteps the
   layout trap below.

If the fixed-88 baseline already fits the deadline, you may not need this at all, and
you keep zero per-block branching. Earn the complexity with a measurement.

### Two gaps to make explicit in the text

1. **Per-block state advance is missing.** "Stateful generation" only covers the
   attack. The load-bearing per-block work is carrying phase and amplitude (the age)
   across callbacks: the per-partial scalars are
   `A_p = amp_p * decay^age * cos(phi_p + age*omega_p)` (and the sin twin), recomputed
   each block from the current accumulated phase/decay. The static table always starts
   at n=0; continuity lives entirely in these scalars.
2. **Name the A/B (sin phi / cos phi) decomposition in "Summation of partials."** The
   multiply is `C @ M`, where `M` is the *static* table and `C` holds the *per-block*
   A/B coefficients at fixed (precomputed) block-diagonal positions. A reader who
   assumes the "mixing matrix" is a static 0/1 key-membership matrix will wonder where
   amp/phase/age enter — they enter through `C`.

### One correctness detail

**Carry `c` clipped.** The envelope level handed to the next block must be the clipped
value in [0,1]: `c_next = clip(m*nsamps + c, 0, 1)`. If you carry the raw line value
(which can exceed 1 during a long sustain), a later note-off would fall from >1 and
the release would be silently delayed.

### Layout trap (triggered by decoupling nsamps from max_nsamps)

You precompute at `max_nsamps` but render at `nsamps < max_nsamps`. If samples are on
axis 1 (`M[:, :nsamps]`), NumPy copies rather than handing BLAS an `lda` stride — a
3-4x penalty. Put **samples on axis 0** so the per-block slice `M[:nsamps]` is
contiguous, or allocate at exact block size. The active-note gather in the section
above naturally produces contiguous data, so these two decisions can be made together.
