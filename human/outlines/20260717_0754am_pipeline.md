## Background and Motivation

The relevant finding is that each NumPy call (involving one or more ndarray arguments)
adds 0.75 microseconds of overhead, in addition to numerical processing.

When scaled to number of steps per partials, number of partials per piano note,
and total number of piano notes voiced, this per-call overhead can grow to
eclipse the numerical processing time.

This is particularly relevant in real time audio synthesis, where the size of each
callback request (nsamps) is in the low hundreds.

Refer to [[claude/deep_research/2026-07-17-additive-synthesis-gemv-batching.md]]
for full analysis and runnable benchmarks.

## Strategy

Unconditionally process all partials for all piano keys, while allowing nsamps
to decouple from the callback's request size.

The ability to decouple nsamps from max_nsamps comes from the Damped Sine and
the Linear Ramp mathematical model.

Reorganize the pipeline into as few NumPy call as possible, leveraging matrix-
matrix and matrix-vector multiplications to the fullest extent.

Minimize any work-saving logic that has a high dynamic line-of-code execution
count in Python, unless the performance gain can be demonstrated.

## Description of the new pipeline

### Pre-computation for Damped Sine

We prepare a list of all partials from all piano keys. This gives the number
of tracks for this pipeline.

Each partial is described with frequency, decay constant (tau), and relative
amplitude.

The frequencies of fundamentals and partials obey either a 12-EDO or JI model,
and additionally JI model can be customized with a tonic.

Frequency and tau are converted into omega and log_decay. From these, the
`sinexp` and `cosexp` parts are initialized, with `iota` given by
`np.arange(max_nsamps)`.

Once initialized, these parameters and the cached Damped Sine components are
cached for the entire application process.

### Stateful generation for Damped Sine

Upon a key press (attack), the piano note velocity is converted into amplitude,
then converted into per-partial amplitude. Each partial gets assigned a uniform
random initial phase.

The Linear Ramp states and parameters are also initialized for the piano note.

### Summation of partials

Before envelope shaping, the partials that make up each piano key are
summed up. This is performed in one matrix-matrix multiplication step,
via a mixing matrix.

After this step, the number of tracks is now equal to the number of piano keys.

### Proposed pipeline for Linear Ramp

Linear Ramp is implemented with a simple model of a line equation,
```
y = clip ( m * x + c, 0.0, 1.0 )
```
Where `c` is the envelope level at the start of each rendering request,
and `m` is either `m_rise` for attack, or `m_fall` for release.

Where `x` is `iota`, or `np.arange(nsamps)`, as usual.

Re-trigger, and continuation of unfinished rise or fall only affect the value
of `c`, which is carried over from the previous callback request.

Values `m` and `c` remain constant across the `nsamps` dimension. Their values
vary across piano keys, based on the Linear Ramp state of each piano key.

In NumPy, `m` and `c` values are said to be *broadcast* on the `nsamps`
dimension.

`clip` is a single NumPy call.

Envelope shaping is accomplished with an elementwise multiplication between
the Damped Sine (after summation of partials) and the Linear Ramp.
