## Memory churn (long-run, N=383)

`analytic B/blk` = bytes of temporaries allocated per block, from known array shapes — the allocator/bandwidth traffic. `transient B/blk` is tracemalloc's single-block peak (Python-object headers only; numpy data buffers use malloc and are untraced). `gc gen0/1/2` = cyclic-GC cycles over the run.

> **Key nuance:** the per-note temporaries hold no reference cycles, so they are freed by **refcounting the instant they go out of scope** — the cyclic collector never triggers (gen0 ≈ 0 even at 34 KB/block). So per-note's churn is **not a GC-pause source**; its cost is the allocator + dispatch traffic, which is already folded into the render time (and is exactly what the batched SoA path — 0 B/block — recovers). The churn matters for sustained allocator pressure and cache behavior, not for stop-the-world GC dropouts.

## Findings

1. **Caching wins.** At N=383, M=16, V=88 the ver5 per-note cached matmul is **12.5× faster** than recomputing sin/exp every block (naive). The gain grows with M.

2. **Batching wins big.** Collapsing only the matmul (still looping the coefficient build per voice) gives 1.9×; the full SoA collapse (one cos+sin over all partials + one GEMV) is **6.4× faster than the per-note path** at V=88/M=16/N=383 (1.29% vs 8.28% of deadline) — confirming the research's 'reduce NumPy dispatch count' thesis. ver5's per-note design leaves that on the table.

3. **Headroom at 88 keys (per-note, N=383):** 9.5% of deadline at p99 → ~924 voices fit. At the N=95 low-latency deadline it is tighter: 30.7% at p99.

4. **Memory churn:** per-note allocates ~33,792 B/block at V=88/M=16 (~3.9 MB/s of alloc/free traffic); batched SoA allocates **0**. But cyclic GC stays quiet on both (gen0=0 vs 0 over the run) — the temporaries have no reference cycles, so refcounting frees them immediately and the collector never fires. So the churn is **not** a GC-pause source; its cost is allocator + dispatch traffic, already inside the render time the SoA path recovers.

5. **The float64 upcast trap is real.** Building `PartialM` arrays as float64 (as the existing test does) makes the f32-requested cache run at **1015.9 µs/block** — matching f64 (872.9) and ~1.4× slower than genuine float32 (718.8). Fix: build the partial arrays as float32.
